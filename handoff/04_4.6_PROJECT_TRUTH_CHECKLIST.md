# 4.6 项目认知对齐清单（与 Cursor / 仓库一致）

> **最近同步**：与当前仓库行为对齐（含 `apply_chat_turn_atomic`、`settings.json`、`PromptInput`、多 Tauri 命令、`app_data_dir` 下 SQLite）。  
> **用途**：接手前逐项核对；若外部文档与本清单冲突，**以本清单 + 源码为准**。  
> **仓库根**：`oclivenewnew`；Rust crate：`src-tauri/`。

---

## 1. 技术栈与版本（勿臆测）

| 认知点 | 事实 |
|--------|------|
| Tauri 主版本 | **`Cargo.toml` 为 Tauri 1.5**，不是 2.x（除非你们已升级并改 manifest）。 |
| 前端 invoke | 以项目 **`package.json`** 里 `@tauri-apps/api` 版本为准选 import 路径。 |
| 数据库 | **SQLx + SQLite**，迁移在 `src-tauri/migrations/`。 |
| 旧 rusqlite | **`src-tauri/src/db` 已删除**；不存在双路径 DB。 |

---

## 2. 三件事原则（架构）

| 认知点 | 事实 |
|--------|------|
| 编排入口 | **`domain/chat_engine/`**（`mod.rs` 内 `process_message`） 的 `process_message(state, req)`，不是 `ProcessMessageResult` 这种未在仓库出现的返回类型名。 |
| 持久化 | **Repository trait** 在 `domain/repository.rs`；实现 **`SqliteMemoryRepository` / `SqliteFavorabilityRepository`** 在 `infrastructure/repositories.rs`，内部持有 **`Arc<DbManager>`**，不是「直接给 `DbManager` impl Repository」。 |
| 业务侧 SQL | **禁止**在 `domain` 里随手 `sqlx::query`；SQL 集中在 **`infrastructure/db.rs`** 与 Repository 实现中。 |
| Tauri 命令 | 定义在 **`api/*.rs`**；**唯一注册**在 **`src-tauri/src/lib.rs`** 的 **`tauri::generate_handler![...]`**；**必须保留** `.setup` 与 **`AppState::manage`**。 |
| `main.rs` | **只调用 `oclivenewnew_tauri::run()`**，**不要**在 `main.rs` 再写一套 `Builder`。 |

---

## 3. 类型与位置（易混）

| 易错认知 | 事实 |
|----------|------|
| `Emotion` 有 Joy、Fearful… | **无**。见 `models/emotion.rs`：**`Happy, Sad, Angry, Neutral, Excited, Confused, Shy`**。 |
| `EmotionResult` 在 `models` | 在 **`domain/emotion_analyzer.rs`**；字段为 **七维 f64**；已有 **`to_emotion()`** → 委托 **`EmotionAnalyzer::get_dominant_emotion`**。 |
| `EmotionResult::default()` | **当前无 `Default`**，测试用**结构体字面量**。 |
| `EventType::Breakup` 等 | **无**。见 `models/event.rs`：**`Quarrel, Apology, Praise, Complaint, Confession, Joke, Ignore`**。 |
| `Role` 有 `personality` 字段 | **无**。性格向量是 **`PersonalityVector`**，来自 **缓存 / DB / `default_personality` 映射**（见 `AppState::get_current_personality`）。 |
| `PersonalityEngine` 只有一个 `evolve` | 实际使用 **`adjust_by_user_emotion`** 与 **`evolve_by_event`** 等组合。 |

## 4. API 与 DTO（与 HTTP 草稿区分）

| 易错认知 | 事实 |
|----------|------|
| 存在 `GET/POST …/api/send_message` | **默认不存在**。联调是 **`invoke('send_message', payload)`**；除非项目**另外**加了 HTTP 层。 |
| 响应字段叫 `response` | **`SendMessageResponse` 里是 `reply`**，不是 `response`。 |
| `emotion` 是字符串如 `"Happy"` | **`emotion` 是 `EmotionDto`**（七维 float），见 **`models/dto.rs`**。 |
| 命令参数顺序 | 当前 **`send_message(req, state)`**（以 **`api/chat.rs`** 为准）。 |

**请求/响应请以 `src-tauri/src/models/dto.rs` 为唯一契约来源。**

---

## 5. 编排层当前行为（勿编造模块名）

| 易错认知 | 事实 |
|----------|------|
| 存在 `FavorabilityEngine::calculate_delta` | **仓库无此模块/方法**；好感度增量来自 **`EventDetector::get_impact_factor`** 等已有逻辑再缩放。 |
| 对话回合如何落库 | **`process_message`** 在适当时机调用 **`DbManager::apply_chat_turn_atomic`**（`ChatTurnTxInput`），在**同一事务**内更新记忆、好感、`events` 等与该回合相关的持久化（实现见 `infrastructure/db.rs`）。不要仅理解为「先 `save_memory` 再单独 `save_event`」一条路径。 |
| bot 情绪从 DB 读 | 当前 **`EventDetector::detect` 的 bot 侧为 `Emotion::Neutral`**（可后续改）。 |
| 立绘/展示用情绪 | **`DbManager::get_current_emotion` / `set_current_emotion`** 存在，供 `load_role` 等与 UI 展示相关的路径使用；与「事件检测里的 bot 情绪」不是同一概念。 |

## 6. 数据库（以迁移为准）

| 易错认知 | 事实 |
|----------|------|
| 表名 `roles` | **无**；运行时好感度在 **`role_runtime`**（含 `current_favorability` 等）。 |
| `role_runtime` 有 `current_personality_json` | **无**；性格历史在 **`personality_vector`** 等表。 |
| 表名 `memories` | 长期记忆表为 **`long_term_memory`**，列以 **`001_init.sql`** 为准（含 `importance`、`weight` 等）。 |
| `favorability_history` 列叫 `value` | 使用 **`delta`**（及迁移中的其他列）。 |
| 好感度更新 SQL | 更新 **`role_runtime.current_favorability`**；实现见 **`DbManager::apply_favorability_delta`**。 |

**唯一 schema 真相来源：`src-tauri/migrations/001_init.sql`。**

## 7. PromptBuilder（签名勿写错）

| 认知点 | 事实 |
|--------|------|
| 入口 | **`PromptBuilder::build_prompt(input: &PromptInput<'_>) -> String`**（见 `domain/prompt_builder.rs`）。`PromptInput` 聚合角色、性格、记忆、用户输入、情绪等上下文，**不是**五个独立位置参数。 |
| 返回值 | **`String`，不是 `Result`**；调用处**不要**对 `build_prompt(...)` 加 `?`。 |
