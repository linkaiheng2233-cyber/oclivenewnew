# 角色包场景系统 — 创作者使用指南

角色包整体结构与「用户身份」配置总览见 [《角色包用户自定义创作者教学》](./CREATOR_ROLE_PACK_CUSTOMIZATION.md)。

本文说明如何在角色包中**配置、扩展与调试场景**，使对话、记忆策略与场景切换行为符合创作意图。**一般情况下，新增场景只需改角色包文件，无需修改程序源码。**

---

## 1. 场景是什么？

**场景（scene）**表示角色与用户互动时所处的**空间或情境**（例如家、学校、公司、游乐园）。  

应用会：

- 按当前场景选择**记忆与策略**（若已配置）；
- 把场景相关说明送进**主对话模型**，让人设语气与环境一致；
- 根据用户台词尝试**自动切换场景**，或由用户在界面里**手动切换**。

---

## 2. 目录结构一览

以角色 ID 为 `mumu` 为例，场景相关文件建议如下组织：

```text
roles/mumu/
├── manifest.json              # 声明 scenes 列表、记忆话题权重等
├── core_personality.txt       # 全局人设（与场景无关）
├── config.json                # 可选：时间衰减等
└── scenes/
    ├── home/
    │   ├── scene.json         # 必填：场景元数据（名称、关键词等）
    │   └── description.txt    # 强烈建议：给主模型读的详细设定
    ├── school/
    │   ├── scene.json
    │   └── description.txt
    └── ...
```

- **`scene_id`**：与文件夹名一致（如 `home`、`school`），建议使用**小写英文 + 下划线**，避免空格与特殊符号，便于跨平台与 API 稳定。

---

## 3. 在 `manifest.json` 中声明场景

在顶层 **`scenes`** 数组中列出本角色所有场景 ID（**推荐显式写出**，便于排序与审阅）：

```json
"scenes": ["home", "school", "company", "park"]
```

说明：

- 程序会合并 **`manifest.json` 中的 `scenes`** 与 **`scenes/` 下实际存在的子目录名`**，再排序、去重。  
- 若两者都为空，会回退到内置的 `default` 逻辑；**正常创作请至少配置一个场景目录 + `scene.json`。**

### 3.1 记忆话题权重 `memory_config.topic_weights`（可选）

若使用「按场景偏好的话题类型」做记忆提示，可为**每个场景 id** 配置一组**话题 → 权重**（权重为 0～1 量级，相对大小有意义即可）：

```json
"memory_config": {
  "scene_weight_multiplier": 1.2,
  "topic_weights": {
    "home": { "日常": 0.75, "学习": 0.15, "工作": 0.1 },
    "school": { "学习": 0.85, "日常": 0.15 }
  }
}
```

- **未出现在此表中的新场景**：不会生成「在本场景更可能聊某话题」那一句提示，**不影响运行**，仅少一条辅助提示。  
- 为新场景补一行 `topic_weights` 即可与现有逻辑对齐。

---

## 4. `scene.json` 字段说明

程序读取 `scenes/<scene_id>/scene.json`，支持的字段如下（均为可选，但**建议至少填写 `name` 与若干 `keywords`**）。

| 字段 | 类型 | 作用 |
|------|------|------|
| `name` | 字符串 | **展示名**（中文即可），如下拉框、系统提示、「当前场景」文案。 |
| `welcome_message` | 字符串 | **切换进该场景时的欢迎语**（优先于 `monologues` 抽一句）。 |
| `keywords` | 字符串数组 | **场景切换规则匹配**：用户话里出现这些词时，更容易被判为要切到本场景；也用于无 `description.txt` 时拼一段简短说明。 |
| `events` | 字符串数组 | 可出现的情节标签，用于同上拼接与氛围参考。 |
| `monologues` | 字符串数组 | 无 `welcome_message` 时，可用作欢迎语/独白的模板池。 |
| `time_windows` | 对象数组 | **可选**：限制**仅在该虚拟时间段内**才允许切入本场景（见下文）。 |

### 4.1 `time_windows`（进阶）

每项为 `{ "start": "HH:MM", "end": "HH:MM" }`（24 小时制）。  

- 若**非空**：只有在**应用内虚拟时间**落在任一窗口内时，**自动切换**才会进入该场景。  
- 若**为空**：不限制时间。  

用于「深夜不开放游乐园」等设定时很有用；若测试阶段切换异常，可先**删掉 `time_windows`** 或放宽范围。

---

## 5. `description.txt`（强烈推荐）

路径：`scenes/<scene_id>/description.txt`  

- **内容**：面向**主对话大模型**的长文设定——环境、人际关系、语气边界、禁忌、典型桥段等，**用自然中文写即可**。  
- **效果**：会进入主提示词中的「场景设定（来自角色包…）」区块；过长会在程序侧**截断**（约数千字量级），避免撑爆上下文。  
- **若没有该文件**：程序会用 `scene.json` 里的 `name`、`keywords`、`events` **自动拼一段短说明**，保证新场景也能用，但细腻度不如手写长文。

**写作建议**：先写 1～3 段总述，再列 bullet 写「在此场景中角色通常如何称呼用户、什么话题多、什么要少提」。

## 6. 场景如何被程序使用（你不必改代码的部分）

### 6.1 场景列表与展示名

- 列表来自 **manifest + 磁盘目录** 合并。  
- 展示名优先 **`scene.json` 的 `name`**；缺失时用内置英文 id 的少量中文映射；再不行则显示 id。

### 6.2 主对话 Prompt

- **当前场景展示名** + **话题提示**（若配置了 `topic_weights`）+ **`description.txt` 或自动拼接的短说明** 会一并进入主对话，模型应在此氛围内回复。

### 6.3 自动场景切换（后端）

1. **规则层**：用户消息中含位移动词（如：去、来、回、到、进、出、逛、前往、回到、来到等），且与某场景的 **keywords / 展示名 / id** 等匹配得分足够，则可能切到该场景。  
2. **模型层**：规则未命中时，会用**小模型**根据「候选场景列表 + 每行摘要」输出 JSON，判断是否切换及目标 `scene_id`。  
3. **摘要**来自角色包：`description.txt` 首行非空行，或「名称 + 首个关键词」。

因此：**keywords 写得越贴近用户口语，自动切换越稳。**

### 6.4 策略插件 `config/policy.toml`（可选）

- 文件位于应用侧 **`config/policy.toml`**（与单个角色包分离）。  
- **`[scene_bindings]`** 可把某个 **scene_id** 映射到 **`conservative`** 或 **`exploratory`** 等策略 profile，影响记忆过滤、默认重要度等。  
- **未在表中出现的 scene_id**：使用 **`default_profile`**，**不报错**。  
- 新场景若需要与「家」同策略，可复制一行并改成新 id 即可。

## 7. 客户端相关行为（便于测试时对照）

- 发送消息后，若后端判定切换成功，可能出现**系统提示**（例如下一回合进入某场景）。  
- 用户句子里含**位移意图**且当前角色**还有其它场景**时，在**本轮回复结束后**可能出现**「要去哪？」**目的地选择；选后会调用切换接口，与顶栏切换一致。  
- 若角色包**只有一个场景**，则不会出现目的地列表——**属于正常现象**。

---

## 8. 新增场景：推荐步骤清单

1. 新建文件夹 `roles/<角色ID>/scenes/<新场景ID>/`。  
2. 编写 `scene.json`：至少 `name`、尽量多的 `keywords`，按需 `welcome_message`、`events`、`monologues`。  
3. 编写 `description.txt`（推荐），写清氛围与互动习惯。  
4. 在 **`manifest.json` 的 `scenes`** 中追加该 id。  
5. （可选）在 **`memory_config.topic_weights`** 里为新 id 加话题权重。  
6. （可选）在 **`config/policy.toml`** 里为新 id 加 `scene_bindings`。  
7. 重新加载角色或重启应用后测试对话与切换。

## 9. 常见问题与排错

| 现象 | 可能原因 | 处理建议 |
|------|----------|----------|
| 前端看不到多个场景 | 角色包只配了一个场景或 manifest 未列出 | 增加目录与 manifest 条目 |
| 自动总切不到某场景 | keywords 与用户说法差距大 | 补充口语化关键词、检查同音字 |
| 某时段永远切不进场景 | 配置了 `time_windows` 且虚拟时间不在窗口内 | 调整窗口或删除该字段测试 |
| 新场景没有「话题偏置」那句 | 未配置 `topic_weights` | 在 manifest 里为该 id 增加一项 |
| 策略与预期不符 | 未在 `policy.toml` 绑定 | 为 scene_id 增加 `scene_bindings` 或使用默认 |

---

## 10. 最佳实践小结

1. **id 稳定**：场景 id 确定后尽量少改，以免用户存档、数据库里的 `current_scene` 对不上。  
2. **keywords 宁多勿少**：覆盖用户可能说的「回家、去公司、到学校」等同义说法。  
3. **description.txt 写「演什么」**：少写系统设定，多写**角色在此场景怎么说话、关心什么**。  
4. **先不配时间窗**：调试通后再加 `time_windows`。  
5. **与全局人设分工**：`core_personality.txt` 管性格；场景文件管**空间与情境**，避免重复粘贴大段相同文字。

---

## 11. 版本与兼容性

- 角色包 **`manifest.json`** 建议随内容变更**递增 `version`**，便于分发与排查。  
- 若应用升级后增加新字段，一般会**向后兼容**：旧 `scene.json` 仍可缺省字段运行。
